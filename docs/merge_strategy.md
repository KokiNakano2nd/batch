# マージ戦略

## そもそもマージとは

`feature` ブランチで作った変更を `main` ブランチに取り込む操作。
取り込み方（＝コミット履歴の残し方）に3種類ある。

---

## 3種類のマージ方法

### A. Merge commit

```
# feature ブランチで2つコミットした場合

main:    A ── B ──────────── M
                    \       /
feature:             D ── E

M = マージコミット（「feature をマージした」という記録）
D, E = feature ブランチのコミットがそのまま残る
```

**特徴：**
- feature ブランチの全コミットが `main` の履歴に残る
- いつ・どのブランチからマージしたかが明確
- コミットが増えるため `git log` が複雑になりやすい

**向いているケース：**
- チームで大きな機能を並行開発していて、ブランチの分岐・合流を可視化したい場合

---

### B. Squash merge

```
# feature ブランチで2つコミットした場合

main:    A ── B ── S

feature:      D ── E
                    ↑
              D と E を1つに圧縮して S として main に追加
              feature ブランチ自体は main の履歴に残らない
```

**特徴：**
- feature ブランチの複数コミットを **1つにまとめて** main に追加
- `main` の履歴が「PR 単位の1コミット」になりすっきりする
- feature ブランチ上の `fix typo`, `wip` などの雑なコミットが隠せる

**向いているケース：**
- 個人開発・小規模チームで `main` の履歴をきれいに保ちたい場合
- 作業中のコミット粒度を気にせず作業したい場合

---

### C. Rebase merge

```
# feature ブランチで2つコミットした場合

main:    A ── B ── D' ── E'

feature:      D ── E
                    ↑
              D, E を main の先頭に「並べ直す」（D', E' は別物）
              マージコミットは作られない
```

**特徴：**
- コミットを1本の直線に並べ直す（マージコミットなし）
- D, E は「新しいコミット」として作り直される（ハッシュが変わる）
- 履歴が一直線で追いやすいが、元のブランチ構造は失われる

**向いているケース：**
- コミット1つ1つを意味のある単位で丁寧に管理しているプロジェクト
- OSS など履歴の品質を重視する場合

---

## 3種類の分類基準

3つのマージ方法は **2つの軸** で整理できる。

### 軸1：履歴の形（線形か・非線形か）

```
非線形（Merge commit）              線形（Squash / Rebase）

main:  A ── B ──── M                main:  A ── B ── C
                ↗ ↗
feature:   D ── E                   履歴が一本道になる

分岐・合流が履歴に残る
```

- **非線形** → いつ・どのブランチからマージしたかが視覚的にわかる
- **線形** → `git log` がシンプルで読みやすい

### 軸2：コミットの粒度（まとめるか・まとめないか）

```
まとめない（Merge commit / Rebase）     まとめる（Squash）

feature:  D ── E                        feature:  D ── E
               ↓                                       ↓
main:    ... D ── E（そのまま残る）       main:    ... S（1つに圧縮）
```

- **まとめない** → 作業の細かい経緯が追える
- **まとめる** → `main` の履歴が PR 単位でシンプルになる

### 2軸で整理した図

```
                      コミットをまとめる？
                       Yes            No
                 ┌────────────┬────────────────┐
履歴が  Yes（非線形） │     ✗      │  Merge commit  │
分岐する？           ├────────────┼────────────────┤
        No（線形）   │   Squash   │    Rebase      │
                 └────────────┴────────────────┘
```

選択基準は「**履歴にどれだけ情報を残したいか**」。
情報が多いほど追跡しやすい反面、履歴が複雑になるトレードオフがある。

## 実務での採用傾向

| 規模・種別 | 採用方式 | 理由 |
|---|---|---|
| 個人・スタートアップ | **Squash**（多数派） | デプロイ頻繁、履歴をシンプルに保ちたい |
| 大規模 OSS・大企業 | **Rebase** | コミット品質が高く、履歴がリリースノートになる |
| レガシー・Git Flow | **Merge commit** | ブランチ構造の可視化を重視、2010年代前半の主流 |

---

## 3種類の比較

| | Merge commit | Squash | Rebase |
|---|---|---|---|
| `main` の履歴 | 複雑になりやすい | **シンプル** | 一直線 |
| ブランチの記録 | 残る | 残らない | 残らない |
| 作業コミットの扱い | そのまま残る | **まとめられる** | そのまま残る |
| コンフリクトリスク | 低い | 低い | 高い |
| 向いている規模 | 中〜大規模 | **個人・小規模** | OSS・中規模 |

---

## このプロジェクトの設定

**Squash merge のみを許可する。**

理由：
- 学習中は `fix`, `wip`, `temp` などの雑なコミットが増えやすい
- `main` の履歴を「PR 単位」で管理することで `git log` が読みやすくなる
- コンフリクトリスクが低く、個人開発に最適

```
# git log のイメージ（Squash 採用後）

* 0f72914 docs: GitHub Flow 運用ルールを追加
* 5089307 ci: PR テンプレートを追加
* 7004013 ci: GitHub Issue テンプレートを追加
* fe02ea3 docs: ディレクトリ構成図を追加
```

PR 1本 = コミット 1行 になり、履歴が追いやすい。

## マージ後のブランチ削除

マージ後は作業ブランチを削除する。
GitHub の設定で「PR マージ後にブランチを自動削除」を有効にすると手動操作が不要になる。

```
# 削除しない場合（ブランチが溜まっていく）
main
feature/add-validation   ← マージ済みなのに残っている
fix/csv-keyerror         ← マージ済みなのに残っている
docs/update-readme       ← マージ済みなのに残っている

# 削除する場合（常にきれい）
main
```
